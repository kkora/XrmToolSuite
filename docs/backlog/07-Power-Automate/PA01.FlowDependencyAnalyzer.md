# Flow Dependency Analyzer — User Stories (Candidate / Backlog)

> **Status:** Candidate backlog — not started (no code).
> **Source:** `all` — `prompt/3.XrmToolBox_ALL_PROMPTS.txt`, Section 7 (Power Automate), item 1. Also in pack file idea #4 'Flow Dependency Analyzer' — same tool.
> **Suggested tag:** `PA01` · **Suggested project:** `XrmToolSuite.FlowDependencyAnalyzer`
> **Overlaps:** Deployment Risk Analyzer already parses flow `clientdata` for connection references and detects draft/unbound flows — this tool is the *dedicated dependency mapper* (table/column/connector/child-flow/custom-API dependency tree + component-impact analysis). Reuse DRA's `clientdata` parser and connection-reference logic rather than re-implementing them. Also overlaps Solution Knowledge Graph (component edges) and Technical Debt Analyzer (hardcoded-value findings).
> **Value/priority (my read):** High — "which flows break if I change this table/column/connector?" is a top pre-deployment question and a solid, feasible slice sits entirely inside Dataverse metadata + `clientdata` (no run-history dependency).

## Notes
- Core data: `workflow` rows where `category = 5` (cloud flows), definition JSON in `clientdata`; `connectionreference` (connectorid, connectionreferencelogicalname); `environmentvariabledefinition`/`environmentvariablevalue`; `solutioncomponent` for solution membership; entity/attribute metadata for table/column resolution.
- Dependency discovery is static parse of `clientdata` JSON — triggers, actions, connector ids, table/column references, child-flow (`Workflow`/`RunFlow`) references, custom API (`customapi`) invocations. No run history needed, so this is fully feasible in the SDK.
- Reuse the shipped Deployment Risk Analyzer `clientdata`/connection-reference parser (lift the shared pieces into `src/Shared/Core/` if not already there); keep the dependency-discovery engine UI-free so it stays liftable into console/CI.
- Read-only: output is a dependency tree, an impacted-flow list for a selected component, risk findings, and an export. No writes.
- Never expose HTTP trigger URLs, connection secrets, or embedded credentials found in `clientdata` — redact.
- Missing-metadata detection (flow referencing a deleted table/column/connection reference) degrades a failed lookup to an informational/warning finding, never throws.

## EPIC-PA01 — Map cloud-flow dependencies and component impact
> **As** an **ALM** engineer, **I want** to see every Dataverse, connector, connection-reference, environment-variable, custom-API, and child-flow dependency of each cloud flow, **so that** I know which flows a proposed change will impact before I deploy.

**Outcome:** a per-flow dependency tree plus a reverse "impacted flows for this component" view, missing-dependency warnings, and a deployment-readiness checklist, exportable for humans and pipelines.

---

## FEAT-PA01-1 — Flow inventory and filtering `[Planned]`
- **US-PA01.1.1** `[Planned]` **As** an ALM engineer, **I want** all cloud flows in the environment listed with owner and status, **so that** I have the full set to analyze.
  - **AC:** Flows load off the UI thread via `Service.RetrieveAll` (`workflow`, `category=5`) with progress and cancellation; owner and state/status show per row.
- **US-PA01.1.2** `[Planned]` **As** a maker, **I want** to filter flows by solution, owner, status, connector, trigger type, and referenced table, **so that** I can focus on a relevant subset.
  - **AC:** Filters apply against parsed dependency data; solution filter uses `solutioncomponent` membership.

## FEAT-PA01-2 — Trigger and Dataverse dependency analysis `[Planned]`
- **US-PA01.2.1** `[Planned]` **As** an ALM engineer, **I want** each flow's trigger type and (for Dataverse triggers) table/operation/filter parsed, **so that** I know what starts it.
  - **AC:** Trigger type and Dataverse table + message (create/update/delete) are resolved from `clientdata`; missing table degrades to a warning finding.
- **US-PA01.2.2** `[Planned]` **As** a maker, **I want** the Dataverse tables and columns each flow reads or writes listed, **so that** I know what a schema change affects.
  - **AC:** Table/column references extracted from action bodies; unresolved logical names are flagged as possible missing metadata.

## FEAT-PA01-3 — Connector, connection-reference, and environment-variable dependencies `[Planned]`
- **US-PA01.3.1** `[Planned]` **As** an ALM engineer, **I want** connectors, connection references, and environment variables each flow depends on listed, **so that** I can confirm they exist in the target.
  - **AC:** Connector ids, `connectionreference` logical names, and environment-variable schema names are resolved and shown; reuses DRA's connection-reference logic.
- **US-PA01.3.2** `[Planned]` **As** a release manager, **I want** flows using direct connections instead of connection references detected, **so that** the package is portable across environments.
  - **AC:** Direct-connection usage is a distinct High finding with remediation.

## FEAT-PA01-4 — Child-flow, custom-API, and HTTP dependencies `[Planned]`
- **US-PA01.4.1** `[Planned]` **As** an Enterprise Architect, **I want** child-flow relationships and custom-API invocations discovered, **so that** I understand cross-flow and pro-code coupling.
  - **AC:** Parent→child flow edges and `customapi` calls are extracted; child flows are linked into the dependency tree.
- **US-PA01.4.2** `[Planned]` **As** an ALM engineer, **I want** HTTP/action dependencies surfaced where metadata is available with any URLs redacted, **so that** external coupling is visible without leaking secrets.
  - **AC:** HTTP actions listed as external dependencies; endpoint URLs and auth values are redacted, never shown or exported.

## FEAT-PA01-5 — Missing/hardcoded metadata risk findings `[Planned]`
- **US-PA01.5.1** `[Planned]` **As** a release manager, **I want** flows referencing deleted/missing metadata or not included in the selected solution detected, **so that** I catch import breakers early.
  - **AC:** Missing-metadata and solution-missing findings carry a Critical/High severity and a remediation note; lookups that fail degrade to Info rather than throwing.
- **US-PA01.5.2** `[Planned]` **As** a Technical-Debt owner, **I want** hardcoded environment URLs, GUIDs, and table names in flow definitions flagged, **so that** flows are environment-agnostic.
  - **AC:** Hardcoded-value findings list the action and the offending literal (redacting anything secret).

## FEAT-PA01-6 — Component impact, dependency tree, and export `[Planned]`
- **US-PA01.6.1** `[Planned]` **As** an ALM engineer, **I want** to pick a component (table, column, connector, connection reference, environment variable) and get the list of impacted flows, **so that** I can scope a change.
  - **AC:** Reverse-lookup panel returns every flow depending on the selected component with severity by flow.
- **US-PA01.6.2** `[Planned]` **As** a DEVOPS engineer, **I want** the dependency tree and findings exported to Excel/PDF/JSON/HTML with a deployment-readiness checklist, **so that** I can attach it to a release and gate a pipeline.
  - **AC:** Export runs off the UI thread; JSON carries the impacted-flow map and a pass/fail readiness flag; secrets/URLs are redacted in every format.
