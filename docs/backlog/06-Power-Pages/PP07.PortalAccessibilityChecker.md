## Notes
- **Static/metadata analysis only.** Per the source's important note, this tool must NOT claim to fully certify WCAG compliance — it detects likely issues in template/content HTML and metadata; runtime/browser testing is still required. The dashboard and every export must carry this disclaimer.
- Data sources (dual schema): `adx_webtemplate`/`mspp_webtemplate` (HTML/Liquid), `adx_webpage`/`mspp_webpage` copy/content where available, `adx_contentsnippet`/`mspp_contentsnippet`, `adx_entityform`/`adx_webform`/`mspp_basicform` (+ steps), `adx_entitylist`/`mspp_list`, `adx_webfile`/`mspp_webfile` image metadata where available.
- HTML/Liquid parsing reuses PP02's parser (extended with an HTML-structure pass): detect `<img>` without `alt`, form controls without associated labels, heading order (`h1..h6`), ambiguous link/button text ("click here"), tables without `<th>`, `<iframe>` without title, missing `lang`/landmark patterns, and inline color/contrast values in CSS where present.
- Contrast checks only where literal CSS color values are available in metadata; computed/runtime styles are out of scope — report as best-effort with a caveat.
- Read-only tool — never edits templates, pages, or files. Output is a remediation checklist. No destructive ops.
- Missing fields/tables degrade to not-applicable/informational, never throw.
- Shared-core reuse: PP05 retrieval layer, PP02 parser, `Service.RetrieveAll`, progress + cancellation, settings round-trip (website), and the shared reporting/export module. Accessibility rules + WCAG mapping stay UI-free and SDK-free for unit testing.

## EPIC-PP07 — Detect static accessibility risks in Power Pages metadata and templates
> **As** a MAKER, **I want** a static checker that flags likely accessibility issues in templates, content, forms, and files, **so that** I can fix obvious WCAG-aligned problems before release — knowing runtime testing is still required.

**Outcome:** an accessibility score, a WCAG-style category summary, template/page findings grids, a form/list findings panel, a remediation checklist, and Excel/PDF/Word/JSON/CSV/HTML exports — across `adx_` and `mspp_` schemas — with a clear "static analysis only, not a certification" disclaimer.

---

## FEAT-PP07-1 — Template & content HTML analysis `[Planned]`
- **US-PP07.1.1** `[Planned]` **As** a MAKER, **I want** images without alt-text patterns detected in templates/content, **so that** I fix missing image descriptions.
  - **AC:** `<img>` tags with no/empty `alt` in template or content HTML → mapped to WCAG 1.1.1, severity Medium; retrieval off the UI thread; both schemas supported.
- **US-PP07.1.2** `[Planned]` **As** a MAKER, **I want** missing/poorly-ordered headings and tables without headers detected, **so that** I improve document structure.
  - **AC:** Missing `h1` or a skipped heading level → Medium (WCAG 1.3.1); data tables lacking `<th>` → Medium; findings cite the template/page and location.
- **US-PP07.1.3** `[Planned]` **As** a MAKER, **I want** ambiguous link/button text and iframe/landmark issues detected, **so that** I fix navigation clarity.
  - **AC:** Links/buttons with text like "click here"/empty → Low/Medium (WCAG 2.4.4); `<iframe>` without title and missing landmark/`lang` patterns → Medium where detectable.

## FEAT-PP07-2 — Form & list accessibility `[Planned]`
- **US-PP07.2.1** `[Planned]` **As** a MAKER, **I want** form fields without labels detected where detectable, **so that** I make forms usable with assistive tech.
  - **AC:** Basic/multistep form fields lacking an associated label (per metadata/HTML) → High (WCAG 3.3.2); works across `adx_entityform`/`adx_webform`/`mspp_basicform`.
- **US-PP07.2.2** `[Planned]` **As** a MAKER, **I want** list rendering accessibility risks flagged, **so that** grids remain navigable.
  - **AC:** Lists whose configured markup lacks header semantics where detectable → Medium with the list name.

## FEAT-PP07-3 — Media & color/contrast `[Planned]`
- **US-PP07.3.1** `[Planned]` **As** a MAKER, **I want** web-file/image metadata checked for missing alt/description where available, **so that** I catch inaccessible media.
  - **AC:** Image web files without descriptive metadata (where the schema exposes it) → Low/Medium; not-applicable where absent.
- **US-PP07.3.2** `[Planned]` **As** a MAKER, **I want** color/contrast risks flagged where literal CSS values exist, **so that** I get an early signal on contrast.
  - **AC:** Inline CSS color/background pairs with computable contrast below threshold → Medium (WCAG 1.4.3) with an explicit best-effort caveat; runtime-computed styles marked out of scope.

## FEAT-PP07-4 — WCAG mapping, score & disclaimer `[Planned]`
- **US-PP07.4.1** `[Planned]` **As** an MGR, **I want** findings mapped to WCAG-style categories with a summary, **so that** I can report against guidelines.
  - **AC:** Each finding carries a WCAG success-criterion reference; a category summary aggregates counts by criterion/severity.
- **US-PP07.4.2** `[Planned]` **As** an MGR, **I want** an accessibility score with a prominent "static analysis, not certification" disclaimer, **so that** results are not overstated.
  - **AC:** Weighted severities produce a 0–100 score with a band; the disclaimer appears on the dashboard and in every export; scoring engine SDK-free and unit-tested.

## FEAT-PP07-5 — Remediation & export `[Planned]`
- **US-PP07.5.1** `[Planned]` **As** a MAKER, **I want** a remediation checklist per finding, **so that** I know exactly what to change.
  - **AC:** Each finding maps to a checklist item with WCAG reference, affected record, and recommended fix.
- **US-PP07.5.2** `[Planned]` **As** an MGR, **I want** the accessibility report exported, **so that** I can share it with stakeholders/auditors.
  - **AC:** Excel/PDF/Word/JSON/CSV/HTML exports come from the shared reporting module, include the disclaimer, and open on demand.

## Definition of Done
- Follows suite conventions; supports `adx_` and `mspp_` schemas; read-only default; static analysis only — no full WCAG certification claim, disclaimer on dashboard and all exports; export formats Excel/PDF/Word/JSON/CSV/HTML.
- All Dataverse access off the UI thread via `RunAsync`/`RetrieveAll`; accessibility rules + WCAG mapping UI-free and unit-tested; missing fields/tables degrade to not-applicable/informational, never throw.
- Testing skeleton under `testing/PortalAccessibilityChecker/` when implementation starts; SDK-free accessibility-rule/scoring logic covered by `testing/UnitTests`.
