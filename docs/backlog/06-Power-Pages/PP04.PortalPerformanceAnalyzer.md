# Portal Performance Analyzer — User Stories (Candidate / Backlog)

> **Status:** Candidate backlog — not started (no code).
> **Source:** `all` — `prompt/3.XrmToolBox_ALL_PROMPTS.txt`, Section 6 (Power Pages), item 4. Not in pack file.
> **Suggested tag:** `PP04` · **Suggested project:** `XrmToolSuite.PortalPerformanceAnalyzer`
> **Overlaps:** Web Template Dependency Analyzer (PP02) already parses Liquid for references/FetchXML blocks — this tool reuses that Liquid/FetchXML parsing but scores complexity/cost rather than dependencies. Depends on a **shared FetchXML analyzer** (filters, link-entity/join count, column count, broad-query detection) that PP04 should factor into Shared/Core so PP02 and future tools reuse it. Do not fork a second Liquid parser.
> **Value/priority (my read):** Medium — valuable proactive guidance, but it is static complexity heuristics (no runtime timing), so findings are directional; best shipped after PP02 supplies the parser.

## Notes
- Data sources (dual schema): `adx_webtemplate`/`mspp_webtemplate` (Liquid + embedded FetchXML), `adx_webfile`/`mspp_webfile` + attached `annotation`/`filesize` for size/type, `adx_webpage`/`mspp_webpage` (template usage, child counts), `adx_entityform`/`adx_webform`/`mspp_basicform`, `adx_entitylist`/`mspp_list`, `adx_sitesetting`/`mspp_sitesetting` (cache settings).
- Shared FetchXML analyzer (net-new, lives in Shared/Core, UI-free): parses FetchXML to count link-entities/joins, selected columns, presence/absence of filters, and flags broad/unfiltered queries. PP02 reuses it for detection; PP04 uses it for cost scoring.
- Static analysis only — this scores configuration complexity, not measured page-load time; findings are risk indicators, and the summary must say no runtime timing was performed.
- Read-only tool — never modifies templates, files, or settings. No destructive ops.
- Missing tables / unretrievable file sizes degrade to informational, never throw.
- Shared-core reuse: `Service.RetrieveAll`, `BatchExecutor` where batching file/annotation retrieves, progress + cancellation, settings round-trip (website + thresholds), and the shared reporting/export module. Liquid + FetchXML analyzers stay UI-free and SDK-free for unit testing.

## EPIC-PP04 — Surface Power Pages performance risks from static configuration
> **As** a PERF engineer, **I want** a tool that scores Liquid complexity, FetchXML cost, and asset weight, **so that** I can find performance risks proactively without waiting for user complaints.

**Outcome:** a portal performance score, a template-complexity grid, a FetchXML-findings panel, a web-file size panel, a page-performance panel, optimization recommendations, and Excel/PDF/JSON/CSV/HTML exports — across `adx_` and `mspp_` schemas.

---

## FEAT-PP04-1 — Template & Liquid complexity `[Planned]`
- **US-PP04.1.1** `[Planned]` **As** a PERF engineer, **I want** each template's includes, snippets, FetchXML blocks, and table queries counted, **so that** I can rank template complexity.
  - **AC:** Counts load via the shared Liquid parser off the UI thread; a template-complexity grid sorts by a composite complexity metric; both schemas supported.
- **US-PP04.1.2** `[Planned]` **As** a PERF engineer, **I want** templates over a complexity threshold flagged, **so that** I target the heaviest first.
  - **AC:** Templates exceeding a configurable threshold → Medium/High with the contributing counts; threshold persists via settings.

## FEAT-PP04-2 — FetchXML cost analysis `[Planned]`
- **US-PP04.2.1** `[Planned]` **As** a PERF engineer, **I want** FetchXML with no filters detected, **so that** I catch unbounded queries.
  - **AC:** The shared FetchXML analyzer flags queries lacking a filter/condition → High with the template and query location.
- **US-PP04.2.2** `[Planned]` **As** a PERF engineer, **I want** FetchXML with many joins/link-entities and too many columns flagged, **so that** I reduce query cost.
  - **AC:** Link-entity count and selected-column count over configurable thresholds → Medium/High; `all-attributes`/broad queries → High.
- **US-PP04.2.3** `[Planned]` **As** a PERF engineer, **I want** the FetchXML analyzer packaged in Shared/Core, **so that** PP02 and future tools reuse one implementation.
  - **AC:** The analyzer is a UI-free static helper over query text with unit tests; PP02 and PP04 both reference it (no duplicate parser).

## FEAT-PP04-3 — Web-file & asset analysis `[Planned]`
- **US-PP04.3.1** `[Planned]` **As** a PERF engineer, **I want** web files analyzed by size and type, **so that** I find heavy assets.
  - **AC:** File size/type retrieved (via attached notes/`filesize`) off the UI thread; large JS/CSS/image files over thresholds → Medium/High; both schemas supported.
- **US-PP04.3.2** `[Planned]` **As** a PERF engineer, **I want** duplicate web files detected, **so that** I can consolidate assets.
  - **AC:** Files matching on name/size/hash where available are grouped and flagged Low/Medium.

## FEAT-PP04-4 — Page, form/list & cache analysis `[Planned]`
- **US-PP04.4.1** `[Planned]` **As** a PERF engineer, **I want** pages with heavy template usage or many child pages/web files flagged, **so that** I spot expensive pages.
  - **AC:** Pages whose template complexity or child/asset counts exceed thresholds → Medium/High in a page-performance panel.
- **US-PP04.4.2** `[Planned]` **As** a PERF engineer, **I want** complex forms/lists and cache-related site-setting issues detected, **so that** I address config-level slowdowns.
  - **AC:** Forms/lists over a field/step/column threshold → Medium; cache settings disabled/misconfigured where expected → Medium/High.

## FEAT-PP04-5 — Score, recommendations & export `[Planned]`
- **US-PP04.5.1** `[Planned]` **As** an MGR, **I want** a portal performance score with a band, **so that** I can track it over releases.
  - **AC:** Weighted severities produce a 0–100 score with a Low/Medium/High band; scoring engine SDK-free and unit-tested.
- **US-PP04.5.2** `[Planned]` **As** a PERF engineer, **I want** an optimization recommendation per finding, **so that** I get concrete actions.
  - **AC:** Each finding carries a plain-language recommendation (add filter, reduce joins, compress asset, enable cache).
- **US-PP04.5.3** `[Planned]` **As** an MGR, **I want** the performance report exported, **so that** I can share it.
  - **AC:** Excel/PDF/JSON/CSV/HTML exports come from the shared reporting module and open on demand.

## Definition of Done
- Follows suite conventions; supports `adx_` and `mspp_` schemas; read-only default; static analysis only (no runtime timing) — stated in the report; export formats Excel/PDF/JSON/CSV/HTML.
- All Dataverse access off the UI thread via `RunAsync`/`RetrieveAll`/`BatchExecutor`; shared Liquid + FetchXML analyzers UI-free and unit-tested; missing data degrades to informational, never throws.
- Testing skeleton under `testing/PortalPerformanceAnalyzer/` when implementation starts; SDK-free FetchXML/complexity/scoring logic covered by `testing/UnitTests`.
