# Portal Health Analyzer — User Stories (Candidate / Backlog)

> **Status:** Candidate backlog — not started (no code).
> **Source:** `all` — `prompt/3.XrmToolBox_ALL_PROMPTS.txt`, Section 6 (Power Pages), item 1. Not in pack file.
> **Suggested tag:** `PP01` · **Suggested project:** `XrmToolSuite.PortalHealthAnalyzer`
> **Overlaps:** Deployment Risk Analyzer already includes some Power Pages readiness checks over the `adx_`/`mspp_` schema (e.g. missing/duplicate site settings, anonymous-access hints). This tool goes broad and dedicated: a full portal health score and issue grid across settings, roles, permissions, pages, templates, snippets, forms, lists, and web files. Reuse the Deployment Risk Analyzer's Power Pages readiness queries; do not duplicate its solution-level risk score.
> **Value/priority (my read):** High — portal config is fragile and scattered across many tables; a one-click health dashboard has broad admin appeal and seeds every other PP tool's data layer.

## Notes
- Data sources (dual schema): classic `adx_website`, `adx_webpage`, `adx_webtemplate`, `adx_pagetemplate`, `adx_contentsnippet`, `adx_sitesetting`, `adx_webrole`, `adx_entitypermission`, `adx_entityform`/`adx_webform` (+ steps), `adx_entitylist`, `adx_webfile`, `adx_redirect`, `adx_websiteaccess`; and enhanced/standard `mspp_website`, `mspp_webpage`, `mspp_webtemplate`, `mspp_pagetemplate`, `mspp_contentsnippet`, `mspp_sitesetting`, `mspp_webrole`, `mspp_tablepermission`, `mspp_basicform`, `mspp_list`, `mspp_webfile`, `mspp_redirect`. Detect which schema is present per environment and normalize into one model.
- Read-only tool — inspects portal metadata only; never creates/updates/deletes records. No destructive ops.
- Missing tables (schema not provisioned, or one model only) degrade to an informational finding, never throw.
- Shared-core reuse: `Service.RetrieveAll` for paging, `BatchExecutor` where multiple retrieves are batched, progress + cancellation via `BackgroundWorker`, settings round-trip for last-selected website, and the shared reporting/export module.
- Analyzers stay UI-free (operate on `IOrganizationService`) so they can be lifted into a console/CI wrapper later.
- Scoring is metadata/static analysis only — no runtime page hits; findings describe configuration risk, not live availability.

## EPIC-PP01 — Give portal admins a single health dashboard for a Power Pages website
> **As** an ADM (Portal Admin), **I want** one tool that inventories a website's configuration and scores its health, **so that** I can find broken config, security gaps, and deployment risks without inspecting each table by hand.

**Outcome:** an overall portal health score, configuration summary cards, a categorized issue grid with severities (Critical/High/Medium/Low/Info), recommendations, and Excel/PDF/Word/JSON/CSV/HTML exports — across both `adx_` and `mspp_` schemas from a live connection.

---

## FEAT-PP01-1 — Website discovery & selection `[Planned]`
- **US-PP01.1.1** `[Planned]` **As** an ADM, **I want** every Power Pages website in the environment listed, **so that** I can pick which one to analyze.
  - **AC:** Websites load via `Service.RetrieveAll` off the UI thread with progress/cancellation, listing both `adx_website` and `mspp_website` records with a schema badge.
- **US-PP01.1.2** `[Planned]` **As** an ADM, **I want** my last-selected website remembered, **so that** I resume where I left off.
  - **AC:** Selection persists via settings round-trip (`LoadSettings`/`SaveSettings`), storing website id + schema only (no credentials).

## FEAT-PP01-2 — Configuration retrieval & summary `[Planned]`
- **US-PP01.2.1** `[Planned]` **As** an ADM, **I want** the selected website's settings, roles, permissions, pages, templates, snippets, forms, lists, and web files retrieved, **so that** I have a complete inventory in one pass.
  - **AC:** All record sets load via `RunAsync`/`RetrieveAll` with progress; retrieval works against whichever schema (`adx_`/`mspp_`) the site uses.
- **US-PP01.2.2** `[Planned]` **As** an ADM, **I want** configuration summary cards (counts per record type), **so that** I can gauge site size at a glance.
  - **AC:** Cards show counts for pages, templates, snippets, settings, roles, permissions, forms, lists, web files with the active schema labelled.

## FEAT-PP01-3 — Structural integrity checks `[Planned]`
- **US-PP01.3.1** `[Planned]` **As** an ADM, **I want** broken page relationships and pages without templates detected, **so that** I can fix rendering failures before release.
  - **AC:** Web pages with a missing parent, missing page template, or dangling reference (`adx_`/`mspp_`) are flagged High with the record name/id.
- **US-PP01.3.2** `[Planned]` **As** an ADM, **I want** inactive pages/templates and broken web-file references detected, **so that** I can clean up dead assets.
  - **AC:** Inactive/statecode-off pages/templates → Medium; web files referenced by pages/templates but absent → High.
- **US-PP01.3.3** `[Planned]` **As** an ADM, **I want** forms/lists referencing inactive Dataverse tables flagged, **so that** I catch broken data bindings.
  - **AC:** A basic/multistep form or list bound to a non-existent or disabled entity → High with the entity logical name.

## FEAT-PP01-4 — Site settings checks `[Planned]`
- **US-PP01.4.1** `[Planned]` **As** an ADM, **I want** missing required site settings detected, **so that** the portal has its baseline configuration.
  - **AC:** A curated required-settings list (per schema) is checked; absent settings → High with the setting name and purpose.
- **US-PP01.4.2** `[Planned]` **As** an ADM, **I want** duplicate/conflicting site settings detected, **so that** I remove ambiguous config.
  - **AC:** Settings sharing name within a website are grouped and flagged Medium, showing the conflicting values.

## FEAT-PP01-5 — Security surface checks `[Planned]`
- **US-PP01.5.1** `[Planned]` **As** a SEC engineer, **I want** anonymous-access risks surfaced at a health level, **so that** I know a deeper scan is warranted.
  - **AC:** Table/entity permissions granting anonymous read/write/delete → Critical/High summary finding that links to the Portal Security Scanner scope (dedicated deep analysis lives there).
- **US-PP01.5.2** `[Planned]` **As** a SEC engineer, **I want** table permissions without parent/child scope configuration flagged, **so that** I catch over-broad grants.
  - **AC:** Permissions using Global scope where a Contact/Account scope is expected → Medium/High per severity note.

## FEAT-PP01-6 — Health score, recommendations & export `[Planned]`
- **US-PP01.6.1** `[Planned]` **As** an MGR, **I want** an overall portal health score with a band, **so that** I can communicate readiness.
  - **AC:** Weighted severities produce a 0–100 score with a Low/Medium/High band on the dashboard; score model is SDK-free and unit-tested.
- **US-PP01.6.2** `[Planned]` **As** an ADM, **I want** a recommendation per finding, **so that** I get concrete remediation steps.
  - **AC:** Each finding carries a plain-language recommendation and the affected record link.
- **US-PP01.6.3** `[Planned]` **As** an MGR, **I want** Excel/PDF/Word/JSON/CSV/HTML exports of the health report, **so that** I can share it.
  - **AC:** All formats come from the shared reporting module and open on demand.

## Definition of Done
- Follows suite conventions; supports `adx_` and `mspp_` schemas; read-only default; export formats Excel/PDF/Word/JSON/CSV/HTML.
- All Dataverse access off the UI thread via `RunAsync`/`RetrieveAll`; analyzers UI-free; missing tables degrade to informational, never throw.
- Testing skeleton under `testing/PortalHealthAnalyzer/` when implementation starts; SDK-free scoring covered by `testing/UnitTests`.
